# Football Forecasting Engine

Leakage-safe probabilistic forecasting for international football matches and
World Cup 2026 reporting.

The project is built around one constraint: every model input must be available
before kickoff. It validates match data, normalizes team names, builds as-of-date
ratings, creates prior-only rolling features, evaluates against baselines, and
publishes calibrated W/D/L probabilities rather than only hard picks.

## Current Status

The promoted match model is documented in [reports/MODEL_CARD.md](reports/MODEL_CARD.md).

| Holdout high-confidence policy | n/a | n/a | n/a | 83.71% | 40.37% |
| Rolling high-confidence policy | n/a | n/a | n/a | 78.55% | 38.91% |

The high-confidence policy is a hard-pick layer only. The probability model
still scores every match, and tournament simulation consumes the full
probability distribution.

The current World Cup report is generated by:

```bash
python scripts/build_world_cup_now_report.py
```

It writes `reports/world_cup_now/` from local snapshots. The report joins current
group-stage results when a local results snapshot exists, but it does not infer
official knockout brackets without a pinned bracket input.

## What Is Implemented

- canonical match schema validation and team-name normalization
- pinned local ingestion with source hashes and reproducible snapshots
- equal-kickoff-safe Elo ratings and prior-only rolling features
- class-prior, Elo, logistic, Poisson, Dixon-Coles, and HistGBM models
- optional CatBoost and LightGBM experiments without making them core deps
- temperature/sigmoid calibration and validation-optimized ensembles
- chronological rolling-origin backtests and frozen holdout evaluation
- Log Loss, multiclass Brier Score, Ranked Probability Score, ECE, and accuracy
- high-confidence abstention policy selected only on rolling validation
- 48-team World Cup fixture reporting and tournament simulation modules
- offline tests for schema, leakage prevention, modeling, and simulation
- Streamlit app for inspecting model reports and World Cup fixture forecasts

## Repository Layout

```text

configs/                     Project, data-source, and feature config

docs/                        Contracts, evaluation notes, and implementation docs
models/*.metadata.json       Public model metadata only
notebooks/                   Reproducible reporting notebooks
reports/                     Model cards and selected evaluation outputs
scripts/                     Thin CLI wrappers around package code
src/football_forecast/       Package code
tests/                       Offline test suite
```

Large or source-specific data under `data/raw/`, `data/interim/`, and
`data/processed/` stays out of Git. Binary model artifacts (`*.joblib`) also stay
local; public metadata and checksums are committed instead.

## Real-Data Workflow

1. Validate the source schema.
2. Normalize team names.
3. Sort matches chronologically.
4. Build as-of-date ratings.
5. Build prior-only rolling features.
6. Train baselines and candidate models.
7. Evaluate with Log Loss, Brier Score, and RPS.
8. Calibrate probabilities on chronological validation data.
9. Simulate tournaments only from probability outputs.
10. Record assumptions and limitations in the model card.

Key commands:

```bash
python scripts/validate_data.py
python scripts/rebuild_all_features.py
python scripts/run_structured_model_search.py
python scripts/finalize_champion_model.py
python scripts/evaluate_selective_accuracy.py
```

## Leakage Rules

- No random train/test splits for historical forecasting.
- Rolling features must use `shift(1)` or an equivalent prior-only mechanism.
- Rating joins must be strictly backward-looking.
- Equal-kickoff matches are batched so same-time results cannot leak.
- Candidate models must be evaluated against simple baselines.
- Hard picks are secondary; calibrated probabilities are the primary output.
